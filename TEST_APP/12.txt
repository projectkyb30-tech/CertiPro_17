---

## üìÇ TASKS TO EXECUTE

---

### 1) BUILD PROCESS
Check:
- how project is built
- scripts
- reproducibility
- environment dependencies
- deterministic builds

If build depends on local hacks ‚Üí risk.

---

### 2) CI PIPELINE (IF EXISTS)
Check:
- does CI run automatically?
- tests executed?
- lint/static checks?
- security scans?
- artifacts generated?
- parallelization?

If CI is weak ‚Üí release quality unstable.

---

### 3) MERGE PROTECTION
Check:
- can broken code be merged?
- are checks mandatory?
- approvals required?

If anyone can push to main ‚Üí high risk.

---

### 4) ENVIRONMENT STRATEGY
Check:
- dev
- staging
- production

Are they separated?
Or everything is one?

---

### 5) CONFIGURATION MANAGEMENT
Check:
- environment variables strategy
- secret injection
- config overrides
- risk of wrong environment usage

If config errors can deploy to prod ‚Üí serious issue.

---

### 6) DEPLOYMENT MECHANISM
Understand:
- manual or automated
- scripts or UI
- containerized or not
- zero downtime possible?
- restart risks?

Manual deploy = scaling pain.

---

### 7) ROLLBACK CAPABILITY
If release fails:
- can we revert fast?
- how long would it take?
- data compatibility concerns?

No rollback = dangerous.

---

### 8) DATABASE MIGRATION SAFETY
Check:
- run automatically?
- backward compatible?
- blocking migrations?
- destructive changes?

Bad migrations = outages.

---

### 9) RELEASE VISIBILITY
Can team know:
- which version is deployed?
- when?
- by whom?

If unclear ‚Üí operational confusion.

---

### 10) FEATURE FLAGS (IF PRESENT)
Check:
- ability to enable/disable features
- safe rollout strategies

Not mandatory but strong maturity signal.

---

### 11) INFRA AS CODE (IF EXISTS)
Check:
- reproducibility
- version control
- environment parity

---

### 12) INCIDENT RESPONSE READINESS
Would team:
- know how to react?
- have playbooks?
- know where logs/metrics are?

If chaos ‚Üí risk.

---

## üß† ADVANCED DETECTIONS

Detect:
- undocumented deploy steps
- manual edits on servers
- secret drift
- dependency on specific engineer
- no staging validation
- hidden cron jobs
