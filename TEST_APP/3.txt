## üìÇ TASKS TO EXECUTE

---

### 1) IDENTIFY ARCHITECTURE STYLE
Determine what exists in reality (not what developers say).

Examples:
- layered
- domain driven
- feature modular
- service oriented
- chaotic mixture

Explain with proof.

---

### 2) MODULE BOUNDARIES
Where does one responsibility end and another begin?

Find violations like:
- UI doing domain work
- controllers with business logic
- services touching database + formatting
- utilities doing everything

---

### 3) DEPENDENCY HEALTH
Analyze import/usage direction.

Danger signs:
- circular dependencies
- high coupling
- everything imports everything
- global god modules

---

### 4) CHANGE IMPACT TEST
If we change:

- data model
- API response
- rule

How many files must change?

Large spread = bad architecture.

---

### 5) SINGLE RESPONSIBILITY CHECK
Big files doing many jobs ‚Üí risky.

---

### 6) ABSTRACTION QUALITY
Are there:
- interfaces?
- contracts?
- boundaries?

Or direct access everywhere?

---

### 7) FEATURE ISOLATION
Can you remove one feature easily?

If no ‚Üí tight coupling.

---

### 8) DUPLICATION OF STRUCTURE
Same logic repeated in multiple modules?

Future nightmare.

---

### 9) EXTENSION SIMULATION
Imagine adding a major feature.

Would it:
- plug nicely?
- or require rewriting core?

---

### 10) OWNERSHIP CLARITY
If bug happens, is it obvious where to look?

If not ‚Üí architecture failure.

---

## üß† ADVANCED DETECTIONS

Find:

- hidden coupling
- anti-patterns
- god services
- fat controllers
- leaking abstractions
- cross-layer violations
